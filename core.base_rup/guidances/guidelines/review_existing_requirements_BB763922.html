Основное описание

У вас могут иметься уже готовые спецификации требований предыдущих или иначе связанных систем - их можно использовать как эталон при ревизии настоящих требований. Или, возможно, Rational Unified Process был внедрен уже после начала проекта.

Пройдитесь по каждому требованию группой из нескольких человек для выявления поведений приложения или их атрибутов. Пропускайте во время прохода пояснительную информацию такую как введения и общие описания системы.

Введите список всех выявленных проблем и назначайте кого-нибудь для решения каждой из них. Если требование неоднозначное, вам может потребоваться сделать некоторые предположения. Ведите список таких предположений, чтобы проверить их правильность во время следующей консультации с заказчиками.

Принимайте во внимание то, что требования составлялись заказчиком. Ищите "неуместные требования", т.е. требования, выходящие за рамки проекта. Если вы не уверены, нужно ли рассматривать какое-либо положение как требование, уточните это у заказчиков.

Оказывается очень эффективно применять для таких проходов по требованиям существующие наброски вариантов использования. Для каждого требования должен быть как минимум один вариант использования. Если в наброске найти подходящий вариант использования не удалось, это может означать, что либо пропущен вариант использования, либо требование неуместно.
