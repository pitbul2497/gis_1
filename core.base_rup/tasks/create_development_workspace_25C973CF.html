Задача: Создание рабочей области разработки
Эта задача описывает, как создавать и использовать Рабочую область разработки.
Дисциплины: Управление конфигурацией и изменениями
Назначение
  •  Рабочая область разработки - это частная область, которая предоставляет среду, в которой сотрудник может вносить изменения в рабочие продукты, так чтобы они не были сразу видны другим.
  • При создании рабочей области составляется представление, которое настраивается так, чтобы любой сотрудник мог 'получить' требуемую версию любого рабочего продукта, которая ему нужна для работы. Стратегии проекта определяют, какие рабочие продукты видны и изменяемы, и для кого.

    Основное описание

    Члены коллектива разработчиков должны знать, над чем они работают, и куда нужно доставить выполненную работу, чтобы она могла быть интегрирована и протестирована вместе с работой других сотрудников. В проекте обычно существует одна общая область и множество частных областей разработки. Частная область разработки позволяет разработчику выполнять свою задачу изолированно.

    Подключаясь к проекту, разработчик получает доступ к определенным рабочим продуктам и рабочей области интеграции. Далее разработчик создает частную рабочую область и заполняет ее содержимым контрольной версии проекта в качестве отправной точки. Контрольная версия проекта создается Администратором конфигурации в среде CM проекта.

    В своей рабочей области разработчик выполняет задачи для созданию рабочих продуктов. Работа сотрудников выполняется согласно установленному порядку. Доставляя измененный набор рабочих продуктов в общую рабочую область проекта, сотрудники делают его доступным для других.

    В общей рабочей области индивидуальная работа интегрируется с работой других сотрудников. Периодически в общей рабочей области создается новая контрольная версия, которая объединяет доставленную работу.

    Разработчики обновляют свои частные рабочие области, чтобы включить набор версий, представленный новой контрольной версией. Периодически по мере увеличения качества и стабильности контрольных версий их атрибут уровня продвижения изменяется, чтобы отразить соответствующий уровень готовности. Примерами атрибутов контрольной версии являются компоновка, тест и выпуск, которые определены как часть стратегий CM проекта.
