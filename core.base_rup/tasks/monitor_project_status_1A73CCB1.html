Задача: Мониторинг состояния проекта
В этой задаче рассмотрен процесс проверки текущего состояния проекта и его сравнения с планами проекта.

        Шаги
        Определение состояние работы
        Цель Сбор информации о качестве и ходе выполнения проекта для оценки текущего состояния 

        На этом этапе руководитель проекта собирает простейшие показатели, указывающие на состояние выполнения проекта и качество продукта. Инструкции по сбору этих показателей приведены в плане измерения проекта.

        Как правило, участники группы проекта на регулярной основе отправляют руководителю проекта отчеты о состоянии выполнения, включающие следующую информацию:

        • Трудозатраты на выполнение конкретных задач
        • Оценка трудозатрат, необходимых для выполнения каждой задачи
        • Список выполненных задач
        • Список опубликованных конечных продуктов
        • Обнаруженные неполадки, требующие внимания руководящих лиц (например, записи проверки). Руководитель проекта может добавить их в список неполадок для дальнейшего рассмотрения и отслеживания.

        Дополнительная информация о показателях приведена в разделе Рекомендации по рабочему продукту: Показатели.

        Вычисление индикаторов состояния выполнения

        Для правильного сравнения состояния выполнения проекта с намеченными планами руководитель проекта должен объединить простейшие показатели, полученные от группы проекта, в полную картину состояния выполнения проекта. Инструкции по вычислению производных показателей ("индикаторов выполнения") приведены в плане изменения проекта.

        Вычисление индикаторов качества

        Помимо мониторинга состояния работы руководитель проекта отвечает за обеспечение качества рабочих продуктов. Показатели качества (также определяются в плане изменения проекта) позволяют получить общую картину состояния проекта по сравнению с запланированными требованиями к качеству.

        Сравнение индикаторов и планов

        Вычислив индикаторы качества и состояния выполнения, руководитель проекта сравнивает их с ожидаемым состоянием проекта в соответствии с планами разработки программного обеспечения и итераций. На данном этапе руководитель проекта проверяет следующие вопросы:

        • Завершены ли все запланированные задачи?
        • Опубликованы ли все запланированные рабочие продукты?
        • Соответствует ли оценка трудозатрат на выполнение текущих задач планам?
        • Соответствуют ли показатели качества (например, число открытых дефектов) запланированным допускам?

        Кроме того, руководитель проекта проверяет индикаторы рисков, связанные с каждой записью из списка рисков, для того чтобы определить необходимость применения стратегий снижения рисков.

        Сравнивая состояние выполнения с планом итераций, руководитель проекта должен учитывать временные ограничения итерации, т.е. если исходный план недостижим, вместо задержки итерации необходимо выбрать функции для исключения из итерации.

        Все зарегистрированные неполадки заносятся в список неполадок (который включается в отчет об оценке состояния). Неполадки, находящиеся в пределах компетенции руководителя проекта, следует устранять непосредственно на этапе Задача: Обработка исключительных ситуаций и неполадок; в некоторых случаях может потребоваться создать профайл неполадки, например, создав запрос изменения для отслеживания неполадки, либо обновив список рисков, если неполадка имеет важное значение или широкий круг заинтересованных лиц.

        Неполадки, требующие внимания контрольного совета проекта, включаются в оценку состояния и передаются на рассмотрение. Как правило, это один из этапов задачи Проверка проекта контрольным советом проекта.
