Описание
Основное описание

Запись проверки представляет собой рабочий продукт оценки, специализированный для фиксации результатов деятельностей проверки.

Краткая схема

1. Идентификация проекта и тип проверки

Идентифицируйте проект и тип проверки; например, проверка кода, проверка трассируемости требований, проверка проекта PRA, проверка планирования проекта.

2. Проверяемые рабочие продукты и цели проверки

Перечислите рабочие продукты, которые будут являться объектом данной проверки, и опишите цели проверки.

3. Участники проверки

Перечислите лиц, которые будут принимать участие в проверке, и их роли в ходе совещания; например, председатель, секретарь, проверяющий, автор.

4. Планирование и местоположение

Укажите расписание проверки. Включите дату, время и место проведения совещания по проверке, а также расписание публикации рабочих продуктов проверки, если они не прилагаются к записи проверки.

5. Обнаруженные неполадки и рекомендации по их устранению

Перечислите все неполадки, обнаруженные в ходе проверки. Проверяющие могут обнаружить:

  • Неполадки рабочих продуктов проверки, которые требуют коррекции; это - изъяны
  • Неполадки проекта, признаки которых идентифицируются из рабочих продуктов проверки
  • Неполадки продукта, признаки которых идентифицируются из рабочих продуктов проверки

Группа проверки может дать рекомендации по устранению неполадок.

6. Состояние элемента действия

Перечислите все элементы действия, являющиеся результатом проверки; они должны быть указаны вместе с определенным владельцем (ответственным за выполнение действия) и целевой датой. Обычно это элементы действий, предназначенные для исправления обнаруженных неполадок. Возможны следующие элементы действий:

Продолжение работы:  Рабочий продукт не считается завершенным и следует продолжить разработку 
Создать распоряжение о работе:  Если неполадка требует планирования новой работы без изменения контрольного рабочего продукта 
Создать запросы на изменения:  Если неполадка требует внесения изменений в контрольный рабочий продукт 

Могут существовать элементы действия из предыдущих проверок данного рабочего продукта. Они должны быть перечислены с указанием состояния (например, открыт/закрыт), владельца и целевой даты или даты закрытия.

7. Вопросы для рассмотрения руководителем проекта

Могут быть обнаружены определенные неполадки или отклонения от нормы, порядок действий для которых не удается согласовать в группе проверки, и которые должны быть отправлены руководству для принятия решения.

8. Дальнейшая проверка

Описывает рекомендации группы по дальнейшей проверке (например, необходима ли дальнейшая проверка) и какая дополнительная информация или данные требуются.

9. Запись работ

Фиксирует работы, проведенные при подготовке и выполнении проверки.

Доводка
Опции представления

Хотя этот рабочий продукт должен использоваться во всех проектах, степень формализма будет различаться в зависимости от таких факторов, как насколько официальной является взаимосвязь между заказчиком и разработчиком, или насколько формально относятся в самой организации-разработчике к соблюдению процесса. Например, в договоре по проекту может быть указано, что проверки должны контролировать: в этом случае рабочий продукт обычно будет рассматриваться как контролируемая запись проверки и ее выводов

Хотя этот рабочий продукт в первую очередь используется для фиксации результатов проверки, его также можно использовать в качестве специализированного ордера на работу или контролирующего документа для управления выполнением проверки. В случае использования с такой целью он выдается участникам проверки перед совещанием.
